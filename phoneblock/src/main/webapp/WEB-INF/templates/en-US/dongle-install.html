Building your own dongle - step by step

Follow the instructions in sequence. At the end, the finished dongle is plugged into the USB port of your router and blocks spam calls silently in the background.

Status: Beta. This guide is tailored to a single, tested hardware type - buy exactly the board from step 1, then every further step will fit. Other boards often work as well, but require different drivers and your own detective work (see link below).

This browser cannot flash the dongle Web flashing only works in Google Chrome or Microsoft Edge on a computer (not on a smartphone). Open this page with Chrome or Edge and try again.

Before you get started

Check briefly that these points are fulfilled - so that there's no disappointment:

  • SIP registrar - either in the router (e.g. Fritz!Box) or directly with the telephony provider, provided this allows external SIP clients ("Registration of an IP telephone possible").
  • 2.4 GHz WLAN within range.
  • 5 V USB power supply - ideally USB port on the router, alternatively any USB power supply unit.

Step 2 - Install USB driver (Windows only)

The dongle uses the CH340 USB serial chip. In order for Windows to recognize it as a COM port, you need the appropriate driver once.

Download CH340 driver (wch-ic.com)

Execute the downloaded file CH341SER.EXE → click on "INSTALL" in the window → done. No restart necessary.

Do you use macOS or Linux?
  • macOS: As of macOS Big Sur, the driver is included in the system - no installation step necessary.
  • Linux: The kernel driver is installed. On Ubuntu, the BRLTTY service can snatch away the CH340 so that you do not see the device file - then uninstalling BRLTTY helps: sudo apt remove --purge brltty. Your user must also be in the dialout group: sudo usermod -aG dialout $USER. Both steps are described in detail in GETTING_STARTED.md.

Step 3 - Insert the dongle into the computer

Plug the dongle directly into a free USB port on your computer. A small LED on the board should light up immediately.

Step 4 - Flash firmware

Click on the button. The browser then asks which connected device it may address. Select the entry with "USB-SERIAL CH340" (Windows) or "USB Single Serial" or "wch.cn" (macOS/Linux) in the name and confirm with "Connect".

A list of all serial devices on the computer appears in the selection dialog. If several CH340 devices are displayed because another device with this chip happens to be connected, remove the others for the moment.

Confirm the security prompt "Erase device and install PhoneBlock Dongle?" with "Install". During the first flash, the entire memory is erased and the firmware is installed - this takes about a minute. Do not disconnect the dongle during this time.

Step 5 - Plug the dongle into the router

When the browser reports "Installation complete!", remove the dongle from the computer and plug it into a free USB port on your router (or any 5 V USB power supply within WLAN range).

Step 6 - Add the dongle to the WLAN (WPS)

The freshly flashed dongle does not yet have any Wi-Fi access data and is waiting for pairing via WPS. The LED flashes quickly (about five times per second) and signals "Pairing mode active".

Now press the WPS button on the router. On the Fritz!Box, this is the "Connect / WPS" button on the top of the device - press briefly until the Wi-Fi LED on the router itself starts to flash. Other routers label the button "WPS" or "Pair". Within a few seconds, the router transmits your Wi-Fi access data to the dongle, which saves it and connects.

Has worked: The LED now flashes slowly (about once per second) - the dongle is in the WLAN, has been assigned an IP and is waiting for the configuration in step 7.

Did not work Check in the router settings whether WPS is activated at all (on the Fritz!Box: WLAN → Security → WPS). If the WPS window was too short, the dongle automatically restarts the pairing - then simply press the WPS button again.

Step 7 - Enter SIP access and PhoneBlock token

In order for the dongle to be able to intercept calls, it still needs two things: your SIP access data (so that it can log on to the router as an additional telephone) and a PhoneBlock token (so that it can query the spam database). You enter both in the dongle's web interface.

Open in the browser:

http://answerbot/

Dongle recognized: http://192.168.178.xxx/ - the button above opens it directly.

If this link does not work (Windows without Bonjour, some mesh WLANs), look in the device list of your router for a device with the name answerbot, make a note of its IP address and call it up in the browser - e.g. http://192.168.178.xxx/.

Enter in the web interface:

  • SIP access data - user name, password and registrar address. For a Fritz!Box, create an additional connection under Telephony → Telephony devices → Set up new device → Telephone (with and without answering machine) → LAN/WLAN (IP telephone). Enter the access data assigned there here and enter fritz.box as the registrar.
  • PhoneBlock token - create an API token in your PhoneBlock settings and copy it into the corresponding field.

Save → the LED is now permanently lit → the dongle is ready for use. Spam calls will now be silently intercepted before your phone rings.

Something is not working, you are using Firefox/Safari, Linux with peculiarities or another board? The detailed instructions with all options, Linux pitfalls and the ESP-IDF toolchain can be found in the repo: GETTING_STARTED.md.

Contribute experience report

The dongle runs reliably here at home on a Fritz!Box - but router models, providers and connection types are a mixed bag, and each combination has its own peculiarities: different SIP dialects, different NAT rules, different ringing delays. So that I can recognize patterns and fix problems in a targeted manner, structured experience reports are worth their weight in gold - regardless of whether everything worked for you or something got stuck.

Setups that are not yet well covered are particularly interesting:

  • Telekom connection with Speedport router
  • Vodafone/Unitymedia cable connection with the supplied provider box
  • 1&1, o2, Deutsche Glasfaser and other providers
  • International connections (Switzerland, Austria, rest of Europe)

Reports are run via GitHub issues - a pre-filled form asks for the most important key data (hardware, router, provider, firmware version, findings) in a structured manner. Other testers can see directly which setups are already covered and which are not.
