# FTC Do Not Call (DNC) Data Import - Design

## Goal

Bootstrap US market spam coverage by importing FTC Do Not Call complaint data into PhoneBlock. The FTC publishes ~12K consumer-reported spam complaints per weekday as downloadable CSV files.

## Approach: Separate Tables + Hash Lookup

FTC data lives in its own tables (not merged into NUMBERS). For v1, only the hash-based lookup path (`GET /api/check`) is extended to include FTC data -- this is the path used by the PhoneBlock mobile app.

## Data Source

**Daily CSV files** from the FTC:

- URL pattern: `https://www.ftc.gov/sites/default/files/DNC_Complaint_Numbers_YYYY-MM-DD.csv`
- Published every weekday by ~noon ET; weekend data on Monday, holidays next business day
- ~12K rows per file
- No API key needed (HTTP GET with browser-like User-Agent)
- Last ~5 weeks of files available

**CSV columns:**

| Column | Example |
|--------|---------|
| `Company_Phone_Number` | `8886749072` (bare 10-digit US) |
| `Created_Date` | `2026-03-05 00:03:04` |
| `Violation_Date` | `2026-03-04 23:01:00` |
| `Consumer_City` | `Antioch` |
| `Consumer_State` | `California` |
| `Consumer_Area_Code` | `925` |
| `Subject` | `Vacation  & timeshares` |
| `Recorded_Message_Or_Robocall` | `Y` or `N` |

Only `Company_Phone_Number`, `Created_Date`, `Subject`, and `Recorded_Message_Or_Robocall` are imported. Consumer location is discarded.

## Database Schema

Three new tables:

```sql
-- Lookup table for FTC subject categories (discovered at runtime, seeded with 15 known values)
CREATE TABLE FTC_SUBJECTS (
    ID       INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    LABEL    CHARACTER VARYING(100) NOT NULL UNIQUE,
    RATING   CHARACTER VARYING(20)  -- PhoneBlock Rating enum name, NULL = unmapped
);

-- Per-phone aggregate stats
CREATE TABLE FTC_NUMBERS (
    PHONE          CHARACTER VARYING(20) NOT NULL,  -- E.164 format: +1XXXXXXXXXX
    SHA1           BINARY(20),                      -- SHA-1 hash of PHONE for hash lookup
    VOTES          INTEGER NOT NULL DEFAULT 0,
    ROBOCALLS      INTEGER NOT NULL DEFAULT 0,
    FIRST_REPORTED BIGINT NOT NULL DEFAULT 0,       -- Unix millis
    LAST_REPORTED  BIGINT NOT NULL DEFAULT 0,       -- Unix millis
    CONSTRAINT FTC_NUMBERS_PK PRIMARY KEY (PHONE)
);
CREATE INDEX FTC_NUMBERS_SHA1_IDX ON FTC_NUMBERS (SHA1);

-- Vote breakdown by subject per phone
CREATE TABLE FTC_REPORTS (
    PHONE          CHARACTER VARYING(20) NOT NULL,
    SUBJECT_ID     INTEGER NOT NULL,
    VOTES          INTEGER NOT NULL DEFAULT 0,
    CONSTRAINT FTC_REPORTS_PK PRIMARY KEY (PHONE, SUBJECT_ID),
    CONSTRAINT FTC_REPORTS_SUBJECT_FK FOREIGN KEY (SUBJECT_ID) REFERENCES FTC_SUBJECTS (ID)
);
```

Sync state stored in existing `PROPERTIES` table as `ftc.sync.last_date`.

## Subject-to-Rating Mapping

The `FTC_SUBJECTS.RATING` column maps FTC subjects to PhoneBlock `Rating` enum values. Stored in the database so it can be adjusted without code changes.

**Default seed data (15 known subjects):**

| FTC Subject | RATING |
|---|---|
| Calls pretending to be government, businesses, or family and friends | G_FRAUD |
| Charities | E_ADVERTISING |
| Computer  & technical support | G_FRAUD |
| Dropped call or no message | C_PING |
| Energy, solar,  & utilities | E_ADVERTISING |
| Home improvement  & cleaning | E_ADVERTISING |
| Home security  & alarms | E_ADVERTISING |
| Lotteries, prizes  & sweepstakes | F_GAMBLE |
| Medical  & prescriptions | E_ADVERTISING |
| No Subject Provided | NULL |
| Other | NULL |
| Reducing your debt (credit cards, mortgage, student loans) | G_FRAUD |
| Vacation  & timeshares | E_ADVERTISING |
| Warranties  & protection plans | G_FRAUD |
| Work from home  & other ways to make money | G_FRAUD |

NULL-mapped subjects contribute to total vote count but don't influence the rating category.

New subjects discovered at runtime are inserted with `RATING = NULL`.

## Import Service

**`FtcImportService`** implements `ServletContextListener`:

- Registers a daily scheduled task via `SchedulerService`
- On each run:
  1. Read `ftc.sync.last_date` from PROPERTIES
  2. Compute next date to import (skip weekends/holidays gracefully -- just retry next day on 404)
  3. Download CSV from FTC URL
  4. Parse with opencsv (already in phoneblock/pom.xml)
  5. For each row:
     - Normalize phone: prepend `+1` to bare 10-digit number
     - Compute SHA-1 hash of the E.164 number
     - Upsert `FTC_SUBJECTS`: insert-if-absent, get ID
     - Upsert `FTC_NUMBERS`: increment VOTES, increment ROBOCALLS if Y, update FIRST_REPORTED/LAST_REPORTED
     - Upsert `FTC_REPORTS`: increment VOTES for phone+subject pair
  6. Update `ftc.sync.last_date` in PROPERTIES
  7. If multiple days behind, repeat for next date
- Bootstrap: on first run (no stored date), import last 5 weeks

**JNDI configuration:**

| Property | Default | Description |
|---|---|---|
| `ftc/enabled` | `false` | Enable/disable FTC import |
| `ftc/schedule/hour` | `6` | Hour to run (UTC) |
| `ftc/schedule/minute` | `0` | Minute to run |

## Query Integration (Hash Lookup Only)

**Modified: `SpamCheckServlet`**

After the existing `resolvePhoneHash(hash)` lookup in NUMBERS:

1. Also query: `SELECT VOTES, ROBOCALLS FROM FTC_NUMBERS WHERE SHA1 = ?`
2. If FTC data exists, add FTC votes to the `PhoneInfo.votes` in the response
3. If the number has no community votes but has FTC votes, return FTC data as the spam signal
4. For rating: query `FTC_REPORTS` joined with `FTC_SUBJECTS` to find the dominant mapped rating

**New MyBatis mapper queries:**

```java
@Select("SELECT VOTES, ROBOCALLS FROM FTC_NUMBERS WHERE SHA1 = #{hash}")
FtcNumberInfo getFtcNumberByHash(byte[] hash);

@Select("SELECT s.RATING, SUM(r.VOTES) as VOTES FROM FTC_REPORTS r " +
        "JOIN FTC_SUBJECTS s ON r.SUBJECT_ID = s.ID " +
        "WHERE r.PHONE = #{phone} AND s.RATING IS NOT NULL " +
        "GROUP BY s.RATING ORDER BY VOTES DESC")
List<FtcRatingInfo> getFtcRatingsByPhone(String phone);
```

## Scope Boundaries (v1)

**In scope:**
- FTC CSV download and import
- Three new database tables with migration
- Hash-based lookup integration in SpamCheckServlet
- JNDI configuration for enable/schedule

**Explicitly out of scope:**
- No changes to NumServlet or SearchServlet (web UI/API number lookup)
- No changes to aggregation tables (NUMBERS_AGGREGATION_10/100)
- No changes to blocklist generation or CardDAV sync
- No changes to mobile API prefix hash lookups
- No writing to NUMBERS table
- No FCC Consumer Complaints integration (future)
